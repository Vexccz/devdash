# DevDash v0.14.0

Port conflicts, meet your new killer.

## What's new

### Ports view
- **New "Ports" sidebar entry** showing every listening TCP/UDP port on your machine with the process holding it (name + PID + local address).
- **DevDash-aware labels** — ports get tagged with the DevDash project they belong to. Green "✓ Project" means DevDash started that dev server itself; yellow "⚠ Project" means the port matches a project's expected dev port but a foreign process is squatting on it (the "why won't `npm run dev` start" scenario).
- **Inline filter** across port, process name, PID, and project name.
- **Click project tag** to jump into that project's detail view.
- **Kill button** on every row. Foreign process → one-click `taskkill /F`. DevDash-managed dev server → confirmation prompt first so you don't nuke your own hot-reload.

### How port detection works
DevDash reads each project's `package.json` plus `vite.config.*` / `next.config.*` to infer the dev port (explicit `port:` config, `--port` in scripts, or framework defaults: Vite 5173, Next 3000, Expo 8081, Uvicorn 8000). That's matched against `netstat -ano` output on Windows or `lsof -iTCP -sTCP:LISTEN` on Unix, then cross-referenced with DevDash's own running child processes to flag managed vs. foreign holders.

## Why

"Port 5173 already in use" is the universal dev frustration tax. Instead of hunting through Task Manager or memorizing `netstat | findstr` incantations, now DevDash tells you exactly who's holding the port and lets you free it in two clicks, with a guardrail if you're about to kill your own running dev server.
